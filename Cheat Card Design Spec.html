Design Specification · Handoff

Quick-Reference Cheat Card System

A complete visual language for compact, dual-purpose reference cards: legible on a phone screen and printable at 8 cards per A4 sheet. This document defines colour, type, layout, iconography, and the card anatomy. It is gear-agnostic — apply it to any subject by swapping the content, not the design.

00

Design Principles

Five rules govern every decision below. When a choice is ambiguous, defer to these.

  • Dual-purpose, single source. One layout must work as a phone-scrollable list and as a print sheet. Design the card; the page is just a grid of cards.
  • Legibility first. Cards are read fast, often small and printed. Monospace for all data values; generous weight contrast; never rely on colour alone to carry meaning.
  • Colour with intent. A near-monochrome base (black / white / grey) with one brand accent. A second alert colour is rationed to a single semantic job.
  • Structure over decoration. No gradients, no shadows, no rounded "app" corners. Flat blocks, hard rules, and a bold edge marker.
  • Systematic variety. Distinguish card types (e.g. scenario vs. universal reference) by inverting the same system — not by inventing a new one.

Card Anatomy

Every card is the same five-part structure. Build it once as a component and repeat.

03 / 16
Category
Scenario Title
FieldPrimary value Fieldsecondary value ToggleON · qualifier ToggleOFF Range18 – 35

Short actionable note with one bold keyword.

  • Edge marker (5px). Full-height bar on the left. Yellow for standard cards, red for reference cards. The instant type-identifier.
  • Black header band. Icon + category label (yellow) + title (white), underlined by a 2px yellow rule. The card number sits top-right, faint.
  • Data rows. A two-column grid: fixed-width grey uppercase label (≈64px) and a monospace value. Row gap is tight (2.5px). Bold the value that matters most.
  • Flag tokens. ON = yellow chip; OFF = red text. The only in-data colour.
  • Tip footer. Divider rule, then one short note prefixed with a ▸ marker. Optional but encouraged — max one line or two.

Iconography

One icon per card, top-left of the header. Build them from geometric primitives only — circles, triangles, rectangles, arcs. Flat fill in currentColor (yellow on the dark header). No outlines-plus-fill mixing unless the shape needs it, no detail, no perspective. A 24×24 viewBox, rendered ~22px. The goal is instant silhouette recognition at thumbnail size, not illustration.

Design the card to a fixed print height (≈67.5mm) and let content fit within it — overflow hidden. The same card markup simply reflows from a scrolling column on screen to a fixed grid in print via a @media print block and @page rule. Force a page break after every 8th card.

Count to the page: size the set to a multiple of 8 so print pages are full (e.g. 16 cards = 2 clean A4 sheets). Put reference cards at the end to fill the last page.
07

Build & Delivery

  • Single HTML file. One self-contained document, no build step. Inline the CSS in a <style> block.
  • Two render targets in one file. Screen styles + a @media print / @page A4 block. Test both before handoff.
  • iOS Safari first. Include viewport-fit=cover, apple-mobile-web-app-capable, and safe-area insets. A sticky "Print A4" button is the only interactive element.
  • Offline option. For field use, embed the fonts so the file is fully self-contained and needs no network.
  • Icons as inline SVG <symbol>. Define once in a hidden defs block, reference with <use>. Keeps the file small and recolourable.
  • Canonical HTML. One <article class="card"> per card; semantic structure so content is easy to edit later.
